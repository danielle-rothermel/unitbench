## Objective

Define one shared fake-data contract for the six Unitbench visualization components (R1–R6) so six different agents build against the same shapes, and the D3 real-data wiring is a pass-through. Done means: the six TypeScript shapes below are approved, after which the `src/fixtures/` module (types + seeded generators + vitest tests) is implemented on branch `r0-fixtures` and PR'd.

Scope is the contract only — no component code, no route changes, and no changes to `tools/unitbench_publish`. Every field is derived from what the real pipeline already produces; three additive gaps are called out explicitly rather than papered over. A few decisions are open for review — see Open Questions at the bottom.

## Grounding: the real data path

Fixtures stand in for the SQL layer, so their keys are the Neon column / JSONB payload keys, verbatim, in snake_case. A component built against a fixture row renders a SQL result row with zero mapping.

## Decisions

<Callout id="decision-bootstrap" tone="decision">

**Bootstrap (R5): raw per-sample arrays; bootstrapping runs client-side.** Neon stores one row per prediction — there is no CI table to pass through, so a precomputed-CI fixture would invent a shape with no real counterpart and bake in N, the resample count, and the confidence level. R5's acceptance ("compare across N") requires re-resampling at different N, which needs raw samples. Cardinality is small (tasks × models × N ≈ 164 × 6 × 3), so a seeded client-side bootstrap is cheap and deterministic.

</Callout>

<Callout id="decision-heatmap" tone="decision">

**Heatmap (R6): raw per-task points; binning runs client-side.** Bin edges and counts are view parameters — R6 requires URL-persisted layout state, matching how the existing heatmap treats axis order (`heatmap-params.ts`) — so pre-binned fixtures would freeze the exact knobs the user is meant to turn. Point cardinality (≈6k) sits under the existing `HEATMAP_MAX_ROWS = 10_000` precedent. The D3 swap is one `GROUP BY task_id, model, target` over `published_predictions` joined to `metrics_json.realized_compression_ratio`.

</Callout>

<Callout id="decision-identity" tone="decision">

**Shared identity: `SampleIdentity` mirroring `published_predictions` keys.** `prediction_id / experiment_id / experiment_kind / task_id / model / sample_index`, snake_case, values shaped exactly as the publisher writes them (`dr_dspy_v1.experiment_id()` and `display_model()` formats; canonical model label per `canonical-model.ts`). One explicit deviation: `sample_index` is required in fixtures while the v1 publisher leaves the column NULL — see Open Questions.

</Callout>
